Two Approaches for glance as a ClawHub Skill

Based on: Pure ClawHub install · Master + per-component SKILL.md · CLI wizard + agent fallback

A

Flat Skill Bundle

One glance/ folder published as one ClawHub skill. Top-level SKILL.md documents all CLI commands. Components live as subdirectories (diary_logger/, mood/, etc.) each with their own SKILL.md as optional companion skills. clawhub install glance gets everything.

B

Scaffold-as-Entrypoint

The published skill is just the scaffold framework + dashboard build. No built-in trackers shipped. Users run clawhub install glance, then the agent scaffolds each tracker on demand via glance scaffold. Built-in components become publishable templates.

Approach A: Flat Skill Bundle (Recommended)

Folder structure after clavhub install glance
~/.openclaw/skills/glance/
├── SKILL.md   ← ALL CLI usage documented here
├── install.sh   ← pip install + glance setup
├── diary_logger/
│  ├── SKILL.md   optional companion skill
│  └── scripts/...
├── mood/
│  ├── SKILL.md
│  └── scripts/...
├── reminder/...
├── mit/...
└── scaffold_component/...

Pros

  • One install, everything works
  • Built-in trackers ready to use immediately
  • Per-component SKILL.md for users who want just one tracker
  • Minimal changes from current structure

Cons

  • Larger bundle size (scripts + migrations for all 4 trackers)
  • Users who only want mood tracking still download everything

Approach B: Scaffold-as-Entrypoint

Published skill = framework only
~/.openclaw/skills/glance/
├── SKILL.md   ← glue: pip install + scaffold usage
├── install.sh
├── scaffold_component/
│  ├── SKILL.md
│  └── templates/...
└── dashboard/build.py

Pros

  • Minimal bundle — just the framework
  • Users choose exactly which trackers they want
  • Built-in trackers can be installed as separate ClawHub skills

Cons

  • More steps: install glance, then scaffold each tracker
  • Scaffold requires pip-dependencies (the glance CLI)
  • Users lose the "everything just works" first experience
